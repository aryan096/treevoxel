# TreeVoxel v2 Design Spec

Date: 2026-03-29

## Overview

v2 focuses on making TreeVoxel more useful for Minecraft builders. Three feature areas:

1. **Improved block palette** — Minecraft block presets with approximate colors, plus fence-mimic branch generation and rendering
2. **Litematica export** — full `.litematic` binary export compatible with the Litematica mod
3. **Expanded slider ranges** — wider parameter bounds for wacky/oversized builds

---

## 1. Data Model Changes

### MinecraftPalette

A new type parallel to the existing `BlockColors`:

```ts
type MinecraftBlockId = string; // e.g. 'oak_log', 'spruce_leaves', 'oak_fence'

type MinecraftPalette = {
  log: MinecraftBlockId;
  branch: MinecraftBlockId;
  fence: MinecraftBlockId;
  leaf: MinecraftBlockId;
};
```

`MinecraftPalette` lives in the Zustand store alongside `BlockColors`. They are kept in sync by the palette UI — selecting a Minecraft block preset sets both the hex color and the block ID.

### Fence sub-type

`'fence'` is an internal voxel sub-type emitted during voxelization for thin branch segments. It is not exposed as a user-facing logical category. The voxel store's `BlockType` union expands to:

```ts
type BlockType = 'log' | 'branch' | 'leaf' | 'fence';
```

Fence voxels carry a connectivity bitmask (4-bit, N/S/E/W) stored alongside the block type in a new parallel map in `VoxelStore`:

```ts
type VoxelStore = {
  layers: Map<number, Map<number, BlockType>>;
  fenceConnectivity: Map<number, Map<number, number>>; // y → pack(x,z) → 4-bit mask
  bounds: { ... };
  count: number;
};
```

### TreeSnapshot

`TreeSnapshot` gains a `minecraftPalette` field so saved and shared configurations carry their full export mapping.

```ts
type TreeSnapshot = {
  presetId: PresetId;
  params: TreeParams;
  blockColors: BlockColors;
  minecraftPalette: MinecraftPalette;
};
```

### Default palettes per preset

Each `Preset` gains an optional `minecraftPalette` field with sensible defaults (e.g. spruce preset defaults to `spruce_log`, `spruce_leaves`, `spruce_fence`).

## 2. Block Palette UI

### MC block dropdown

Each logical type (log, branch, leaf) in the palette panel gets a **Minecraft block dropdown** above its existing color picker. Selecting a block:
- Sets the `MinecraftPalette` entry for that type
- Auto-fills the hex color with the block's approximate color (from `MC_BLOCK_PRESETS`)
- The user can still override the hex color manually after selecting

### Fence block row

The `fence` sub-type gets its own MC block dropdown (defaulting to the matching wood fence, e.g. `oak_fence`). It is displayed inline with the branch row under a small "Branch details" disclosure element. It does not get a separate color picker — fence voxels inherit the branch color in the viewport.

### MC_BLOCK_PRESETS

A new file `src/core/minecraftBlocks.ts` contains a curated list:

```ts
type McBlockPreset = {
  id: MinecraftBlockId;
  label: string;
  approximateHex: string;
  category: 'log' | 'branch' | 'leaf' | 'fence';
};
```

Representative entries per category:

- **log**: `oak_log`, `spruce_log`, `birch_log`, `jungle_log`, `acacia_log`, `dark_oak_log`, `mangrove_log`, `cherry_log`, `bamboo_block`, `stripped_oak_log` (and stripped variants)
- **branch**: same wood log set (builders often use stripped logs for branches)
- **leaf**: `oak_leaves`, `spruce_leaves`, `birch_leaves`, `jungle_leaves`, `acacia_leaves`, `dark_oak_leaves`, `mangrove_leaves`, `cherry_leaves`, `azalea_leaves`, `flowering_azalea_leaves`
- **fence**: `oak_fence`, `spruce_fence`, `birch_fence`, `jungle_fence`, `acacia_fence`, `dark_oak_fence`, `mangrove_fence`, `cherry_fence`, `bamboo_fence`

## 3. Fence-Mimic Branch Generation and Rendering

### Voxelization

In `src/core/voxelize.ts`, a new helper `voxelizeFenceSegment()` is called when a branch segment's radius falls below `minBranchThickness`. It:

1. Places a `'fence'` voxel at each step along the segment
2. After all fence voxels are placed, runs a connectivity pass: for each fence voxel, checks its 4 horizontal neighbors (N/S/E/W) in the voxel store and encodes the result as a 4-bit mask
3. Stores the mask in `VoxelStore.fenceConnectivity`

### Render buffer

`buildRenderBuffer` in `src/core/renderBuffer.ts` gains a fence connectivity pass after voxelization. The `RenderBuffer` gains two new instanced arrays:

```ts
type RenderBuffer = {
  matrices: Float32Array;       // existing — full-cube voxels (log, branch, leaf)
  types: Uint8Array;            // existing
  colors: Float32Array;         // existing
  count: number;                // existing

  fencePostMatrices: Float32Array;  // center post per fence voxel
  fencePostColors: Float32Array;    // branch color, one per fence post
  fencePostCount: number;
  fenceArmMatrices: Float32Array;   // one entry per active arm (up to 4 per voxel)
  fenceArmColors: Float32Array;     // branch color, one per arm instance
  fenceArmCount: number;
};
```

### Fence geometry (VoxelMesh.tsx)

Fence voxels are rendered with two separate `InstancedMesh` instances:

- **Post mesh**: a narrow box geometry (0.25 × 1.0 × 0.25 in block units), one instance per fence voxel, colored with the branch color
- **Arm mesh**: a thin horizontal bar geometry (0.5 × 0.25 × 0.125 in block units), rotated per direction, one instance per active arm, placed at ⅜ and ¾ height — matching Minecraft fence proportions

Full-cube voxels (log, branch, leaf) continue using the existing `InstancedMesh` path.

## 4. Litematica Export

### NBT library

Use `nbtify` for browser-compatible NBT binary encoding. If `nbtify` does not cover the full litematic schema, implement a minimal hand-rolled NBT encoder for the required compound structure only.

### .litematic schema

A single-region `.litematic` export:

- `Metadata` compound: name (`"TreeVoxel Export"`), author (`"TreeVoxel"`), dimensions (from `VoxelStore.bounds`), total block count, timestamp
- One `Region` matching the tree's bounding box
- `BlockStatePalette`: array of Minecraft block state strings derived from `MinecraftPalette`
  - Log blocks: `minecraft:oak_log[axis=y]` (always axis=y for tree trunks)
  - Fence blocks: `minecraft:oak_fence[north=true,south=false,east=true,west=false]` — directional properties from the connectivity bitmask
  - Leaf blocks: `minecraft:oak_leaves[persistent=true,distance=1]`
- `BlockStates`: packed long array encoding palette index per position

### Export function

New `exportLitematic(store: VoxelStore, palette: MinecraftPalette): Uint8Array` in `src/core/export.ts`. The existing `exportJSON` and `exportTextGuide` functions are unchanged.

### UI

A new "Export Litematica (.litematic)" button in the toolbar. Triggers `exportLitematic`, wraps the result in a `Blob`, and downloads it.

## 5. Expanded Slider Ranges

Changes to `PARAMETER_DEFS` in `src/core/parameters.ts`:

| Parameter | Current Range | New Range |
|---|---|---|
| `height` | 5–80 | 5–200 |
| `crownWidth` | 2–40 | 2–120 |
| `trunkBaseRadius` | 1–6 | 1–20 |
| `primaryBranchCount` | 2–16 | 2–40 |
| `branchOrderDepth` | 1–4 | 1–6 |
| `leafClusterRadius` | 1–5 | 1–15 |
| `randomSeed` | 0–99999 | 0–999999 |
| `crownDepth` | 0.1–0.95 | 0.05–1.0 |
| `branchLengthRatio` | 0.2–1.0 | 0.1–2.0 |

All other parameters (ratio/density sliders) retain their current ranges.

No UI changes are needed — slider bounds update automatically from `PARAMETER_DEFS`.

---

## Out of Scope for v2

- Per-voxel block assignments (different wood types on trunk vs scaffold)
- Space colonization branch refinement (deferred from v1)
- AI or image-based generation
